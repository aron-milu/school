PART A

API Specification — 20 Groups, 75 Endpoints

1
Authentication & User Management

Handles user authentication, registration, password management, and profile operations. All other endpoints require a valid JWT token.

⚠ Implementation Rules

  • Login must validate against users table with password_hash verification
  • JWT must include: user_id, role, school_id, is_active
  • If must_change_password = true, login returns a flag forcing password reset before any other API access
  • Registration for students/guardians requires a valid school_id reference
  • Role field is immutable after creation — only super_admin can change roles
  • Failed login attempts must be rate-limited (max 5 per 15 minutes per email)
#MethodEndpointDescription
1.1POST/auth/loginAuthenticate by email/phone + password. Returns JWT + user profile with role, education_level, school_id. Checks is_active flag.
1.2POST/auth/logoutInvalidate current session/token. Client discards stored credentials.
1.3POST/auth/registerRegister new user (student, teacher, guardian). Validates against institution records. Auto-assigns education_level from school type.
1.4POST/auth/reset-passwordInitiate password reset. Sends reset link via email/SMS to registered contact.
1.5PUT/auth/change-passwordUpdate password. Required for must_change_password first-login flow. Must validate current password.
1.6GET/users/{id}Get full user profile with role-specific fields (education_level, school_id, guardian links).
1.7PUT/users/{id}Update profile: full_name, phone_number, email. Role and education_level are immutable.

Institution Management

CRUD for educational institutions. Each institution has a type that determines terminology and feature behavior across the platform.

⚠ Implementation Rules

  • Maps to existing schools table
  • school_id (text) is the human-readable code (e.g. LUB-2026), id (uuid) is the primary key
  • Creating a school auto-creates the admin user account and sends credentials
  • Status transitions: trial → active → suspended → expired. Cannot skip steps.
  • School admin can only access their own school_id data
  • Stats endpoint must compute aggregates from students, teachers, classes tables (not stored)
#MethodEndpointDescription
2.1GET/institutionsList all institutions. Super admin only. Filter by type, district, status, tier. Paginated.
2.2POST/institutionsCreate institution. Includes name, type, regNo, district, tier, admin assignment. Super admin only.
2.3GET/institutions/{id}Get institution with computed stats: student count, staff count, active classes, subscription status.
2.4PUT/institutions/{id}Update institution profile: motto, district, tier, settings. School admin (own) or super admin.
2.5PATCH/institutions/{id}/statusChange status: active, suspended, trial, expired. Super admin only.
2.6GET/institutions/{id}/statsAggregated dashboard stats: total students (active/inactive), staff, classes, avg attendance %, income/expense totals.

R2
Cross-Role Workflow Rules

These workflows define how data flows between roles. Each workflow must be implemented as an atomic operation or event-driven chain.

W1: Assignment Lifecycle (Teacher → Student → Guardian)

  • Step 1: Teacher creates assignment → INSERT into assessments table → Notification sent to all students in class
  • Step 2: Student views assignment → READ from assessments (filtered by class_id)
  • Step 3: Student submits → INSERT into student_progress (status='submitted', submitted_at=now()) → Notification to teacher
  • Step 4: Teacher grades → UPDATE student_progress (score_obtained, teacher_remarks, status='graded', graded_at=now()) → Notification to student AND guardian
  • Step 5: Guardian views grade → READ from student_progress (via guardian_students link)

W2: Student Enrollment (Admin → Teacher → Student)

  • Step 1: Admin enrolls student → INSERT users + students records → Assign to class
  • Step 2: Student appears in teacher's class roster → Teacher can view student details
  • Step 3: Student gets access to class assignments, materials, and communities
  • Step 4: Guardian links child via regNo → INSERT guardian_students → Guardian gets read access to child's data

W3: Material Upload & Approval (Teacher → Admin → Student)

  • Step 1: Teacher uploads material → INSERT courses (verification_status='pending') + Upload file to Storage
  • Step 2: Admin reviews → UPDATE courses (verification_status='approved' or 'rejected') → Notification to teacher
  • Step 3: If approved, material appears in student's study materials list (filtered by verification_status='approved')

W4: Attendance Alert (Teacher → Admin → Guardian)

  • Step 1: Teacher records attendance → INSERT attendance_records
  • Step 2: System checks: if student attendance rate drops below 80%, trigger notification
  • Step 3: Notification sent to school admin AND linked guardian

W5: Institution Setup (Super Admin → School Admin → Teachers → Students)

  • Step 1: Super admin creates institution → INSERT schools + admin user account
  • Step 2: School admin logs in with must_change_password=true → Forced password reset
  • Step 3: School admin creates classes, enrolls teachers and students
  • Step 4: Teachers and students get access to their respective dashboards

R4
Database Schema Gaps & Required Migrations

The existing schema has 11 tables. The following tables and columns are missing and must be created before the corresponding APIs can function.

Critical Gaps (APIs blocked without these)

  • conversations table — Blocks: Messaging API (11). Fields: id, type (direct/group), group_name, created_by, school_id, created_at
  • conversation_members table — Blocks: Messaging API (11). Fields: id, conversation_id, user_id, last_read_at, created_at

⚠ Critical RLS Rules

  • NEVER use USING (true) in any policy — this defeats RLS entirely
  • Every table MUST have RLS enabled: ALTER TABLE x ENABLE ROW LEVEL SECURITY;
  • Default state after enabling RLS: NO access. Policies must explicitly grant access.
  • Always use auth.uid() not current_user for user identification
  • SELECT policies use USING; INSERT use WITH CHECK; UPDATE use both; DELETE use USING
  • Do NOT use FOR ALL — separate into individual SELECT/INSERT/UPDATE/DELETE policies
